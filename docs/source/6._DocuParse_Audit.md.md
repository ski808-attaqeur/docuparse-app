# audit.md — Reality Check

> **Purpose:** A candid, no-hype assessment of the DocuParse plan before you sink time into it.
> Read this alongside the optimistic docs (`PRD.md`, `TRD.md`). This one's job is to poke holes.

## TL;DR

The plan is sound and buildable, but the **stated MVP is too big**. "All document types × all three goals (extraction + OCR + Q&A)" is really three products stapled together. For a solo/internal build, ship a thin vertical slice on **one document type** and **one goal** first, prove it works on *your* real documents, then widen. The biggest risks aren't technical — they're **extraction accuracy**, **LLM cost/latency**, and **the maintenance tail** of supporting every format.

---

## 1. Scope reality

| The plan says | The reality |
|---|---|
| Handle all doc types (PDF, scans, DOCX, XLSX, images) | Each format is its own long tail of edge cases. "Support XLSX" quietly means merged cells, multiple sheets, formulas, charts. Budget more than you think. |
| Do extraction + OCR + semantic Q&A | These are three subsystems with different failure modes. Building all three before validating any one spreads you thin. |
| One MVP | Realistically this is a phased program. Treat the PRD's "MVP" as the *vision*, not the *first release*. |

**Recommendation:** First release = **PDF + images → text/OCR → structured extraction for ONE schema you actually need (e.g. invoices)**, with a review UI. Add corpus Q&A and Office formats *after* that slice earns its keep.

## 2. The accuracy problem (the real one)

LLM + OCR extraction is **probabilistic**, not deterministic. Expect:
- **OCR errors** on low-quality scans, skew, handwriting, stamps, low DPI. Tesseract is decent on clean text and mediocre on messy real-world scans. Vision models are better but cost per page.
- **LLM extraction mistakes:** transposed/hallucinated numbers, wrong date when several appear, mis-parsed line items, confident-but-wrong values.
- **No free lunch on confidence:** model self-reported confidence is only loosely calibrated. Treat it as a hint, not a guarantee.

**Consequences for the design (already baked into PRD/TRD, keep them):**
- The **human review step is not optional** — it's the core safety mechanism. Design for fast correction, not for "trust and forget."
- You need a **golden dataset + eval harness from day one** (see `testing.md`), or you'll have no idea whether a prompt change helped or hurt.
- Set expectations: this tool *accelerates* data entry; it doesn't *eliminate* the human for anything that matters (money, legal).

## 3. Cost & latency reality

Every document can trigger: classification (cheap), extraction (Sonnet, sometimes Opus), embedding (cheap-ish), and each Q&A (Sonnet + retrieval). This adds up.

- **Per-document LLM cost is real and variable** with page count and whether you send page *images* (vision) vs. text. A big scanned batch can get expensive fast.
- **Latency:** vision/OCR on large docs is seconds-to-minutes, which is exactly why the async worker design is right. But it means "instant" is off the table for heavy docs — set UI expectations.
- **Mitigations (already in TRD — enforce them):** model tiering (Haiku→Sonnet→Opus), cap pages/tokens per doc, prefer text over vision when text is clean, cache embeddings, and **log token counts per call** so you can actually see spend. Add a simple cost log early; retrofitting it is annoying.

> Verify current pricing at https://docs.claude.com before assuming any number. Do a quick cost estimate on a realistic batch (e.g. 100 typical docs) before committing to a workflow.

## 4. Build vs. buy

Before writing a custom parser for every format, honestly weigh managed options:
- **Managed doc parsing** (e.g. AWS Textract, Google Document AI, Azure Document Intelligence, LlamaParse, Reducto, Docling) can do OCR/layout/table extraction better than a hand-rolled Tesseract pipeline, sometimes cheaper than your time.
- **Trade-off:** managed services mean sending documents to a third party (a privacy consideration for personal/financial data) and a dependency, but they save enormous effort on the format long tail.
- **Docling / Unstructured** are open-source unified parsers you can self-host — a middle path worth prototyping before committing to per-format code.

**Recommendation:** Prototype your *actual* documents through 1–2 managed/open parsers for an afternoon. If they're good enough, don't build a parser zoo. Spend your energy on the extraction schemas, review UX, and Q&A — the parts that are actually your product.

## 5. Privacy & data handling

- These documents likely contain **PII / financial data**. The plan sends text (and sometimes page images) to **external LLM/embedding APIs**. Decide consciously whether that's acceptable.
- Options if it isn't: local OCR (Tesseract) + local embeddings (`sentence-transformers`) + a **local LLM** for extraction/Q&A. This trades accuracy and effort for privacy. The architecture supports swapping these — but a local LLM at usable quality has its own hardware/setup cost.
- At minimum: keep data self-hosted, gitignore `.env`, don't log document contents or secrets, and review the LLM provider's data-retention terms.

## 6. Complexity / maintenance tail

Even "internal" tools accrete work:
- **Infra to run:** Postgres, Redis, workers, object storage, the API, the SPA. Docker Compose keeps it sane, but it's still 5+ moving parts to keep alive.
- **Dependency churn:** parsing libs, OCR, and model APIs all change. Model names/pricing shift (see the export-control episode around mid-2026 as a reminder that provider availability can change under you). Pin versions; revisit deliberately.
- **The 80/20 trap:** the demo works on clean docs in an hour; the last 20% (bad scans, weird layouts, giant spreadsheets) is where the real time goes.

## 7. Honest effort estimate (solo)

Rough, order-of-magnitude, assuming part-time solo work:

| Milestone | Ballpark |
|---|---|
| Thin slice: upload → text/OCR → 1 extraction schema → review UI | ~2–4 weeks |
| Add semantic search + single-doc Q&A with citations | +1–2 weeks |
| Add corpus Q&A + remaining formats (DOCX/XLSX) + export polish | +2–4 weeks |
| Evals, cost logging, hardening, edge cases | ongoing |

These are guesses to calibrate expectations, not commitments. The variance is dominated by how messy your real documents are.

## 8. What could sink this

1. **Skipping the golden set / evals** → you tune blind and regressions creep in.
2. **Trusting extractions without review** → silent bad data, worse than no tool.
3. **Boiling the ocean** → building all formats/goals at once, shipping nothing usable.
4. **Ignoring cost until the bill** → no per-call logging, surprise spend.
5. **Reinventing parsers** the format long tail eats your time better spent on UX/schemas.

## 9. Recommended path (de-risked)

1. **Pick one document type you actually have a stack of** (invoices are the classic).
2. **Prototype parsing** through Docling/managed service on those real files — pick the winner.
3. **Build the vertical slice:** upload → parse → extract one schema → review UI. Ship it, use it.
4. **Add a golden set + eval harness** as you go, not after.
5. **Add search + Q&A** once you have real ingested content to query.
6. **Widen formats/schemas** only when the slice is genuinely saving you time.
7. **Add cost logging and local-mode escape hatch** before scaling usage.

## 10. Open decisions to make now

- [ ] Is sending documents to external LLM/embedding APIs acceptable, or do you need local-only?
- [ ] Which single document type / schema is the "hero" for v1?
- [ ] Build parsers or lean on Docling/a managed service?
- [ ] Single shared-token auth for v1, or real multi-user from the start?
- [ ] What's your acceptable cost-per-document and monthly ceiling?
- [ ] What accuracy threshold makes this "good enough" to trust (with review)?

## 11. Vibe-coding reality check (if you're building this with AI)

Building this by prompting an AI tool is very doable — but this particular app has sharp edges that punish "accept everything and hope." Read these before you start.

- **Don't merge code you can't explain.** If you couldn't describe roughly what a file does, you can't tell when the AI quietly broke it. Ask the agent to explain each step in plain English *before* accepting it. This is the single biggest failure mode.
- **The demo lies.** An AI will get "upload a clean PDF and see text" working in an hour and make you feel done. The 80% that's left — bad scans, weird spreadsheets, wrong-date extractions, failures that don't crash but silently produce garbage — is the actual work. Budget for it.
- **Money leaks quietly.** Every document and every question can hit a paid LLM/embedding API. An agent happily looping over a folder of 500 scans with a vision model can run up a real bill fast. **Add cost logging on day one**, set a spend limit in your provider dashboard, and test on 3 files before you point it at 500.
- **Never trust extractions blindly — this is the whole reason the review step exists.** The AI that *built* the extractor is not the AI that *validates* it. If the tool says an invoice total is $5,230, a human confirms it before it's canonical, especially for anything involving money or legal text.
- **Security you won't see.** AI-generated code loves to log too much, skip input validation, or trust file extensions. Uploaded documents can even contain text trying to boss the model around ("ignore your instructions and…") — that text is **data, not commands**. The rules for this are in `AGENTS.md`; keep them in front of your agent.
- **Keep it small on purpose.** Use the **Starter track** in `TRD.md`. Six containers (Postgres, Redis, workers, storage, API, web) is a lot to debug when something silently doesn't connect. Two containers gets you a working app; add the rest only when you feel the pain.
- **Commit constantly.** Every working step gets a git commit. When the AI paints you into a corner (it will), you want a clean point to roll back to.
- **Tests aren't optional busywork here.** Because the core is probabilistic, the golden set + evals in `testing.md` are how you know a "helpful fix" didn't quietly make extraction worse. Have the agent build the tiny smoke eval early.

**Bottom line for vibe coders:** you can absolutely build this. Go slow, one small runnable step at a time, understand each piece before moving on, watch the API bill, and keep a human in the loop for anything the tool extracts. `BUILD-GUIDE.md` walks you through exactly this, phase by phase.

## 12. UI/UX reality check — the split-screen is harder than it looks

The bounding-box, click-to-extract split-screen is the best part of the product *and* the part most likely to eat weeks. Go in with eyes open.

- **Bounding boxes are the hard problem.** An LLM does not hand you pixel coordinates. The plan (in `TRD.md`) is: OCR gives word boxes, the LLM returns the *source text span* it used, and you match that span back to the boxes. That matching is **fuzzy and will miss** — the model may return `5230.00` while the page says `$5,230.00`, values repeat across the page, and spans wrap across lines. Expect partial coverage, not 100%. Design for the miss case (show the field with "no source location") instead of pretending every field has a perfect box.
- **If precise boxes are non-negotiable,** don't hand-roll it — that's the moment to use a layout/document-AI model or service that returns coordinates natively (Textract, Document AI, Docling, etc.). Weigh that against the privacy trade-off of sending documents out.
- **Click-to-extract is a second mapping problem.** A user-drawn region → which words are inside it → coerce to the field's type (a dragged box over "05/01/2026" must become a valid date). Doable, fiddly, lots of edge cases.
- **The polished parts add up:** pan/zoom, an overlay that stays aligned as you zoom, an editable spreadsheet-grid for line items (add/remove rows, keep them linked to the source), keyboard navigation, and a responsive fallback. Each is modest; together they're the bulk of the frontend.
- **Multimodal grounding doesn't fit the text-box model.** Highlighting a *chart* or *formula* and showing its extracted interpretation is a different mechanism than word-box highlighting. Treat "extract charts/formulas" as its own later phase, not part of the v1 split-screen.

**Phasing that keeps you sane (build in this order):**
1. Source render + grouped, editable fields + confidence cues + Approve/Reject. **No boxes yet.** This alone is a usable tool.
2. **One-way** linking: click a field → highlight its box on the source (needs word boxes + span matching).
3. **Two-way:** click-to-extract from the source back into a field.
4. Editable line-item tables linked to the source.
5. Pan/zoom polish, keyboard-driven review loop, responsive tabs.
6. (Later) multimodal element highlighting.

Ship after step 1, add the rest as it proves its worth. Don't build step 5 before step 1 is something you actually use.

## 13. Integrations reality check (Approve → push)

"Approve sends the data to my CRM/ERP/Sheet" is a great UX promise and a deceptively large surface area. Be deliberate about scope.

- **Not all destinations are equal.** A **generic signed webhook** and **Google Sheets** are genuinely easy and cover most personal/internal needs. **CRMs and ERPs** are each their own integration project — auth (OAuth), field mapping, rate limits, sandbox vs. production, and APIs that change. Don't promise "works with my ERP" until you've read that specific ERP's API docs.
- **Start with webhook + Sheets.** They unblock 80% of the value. Add a specific CRM/ERP only when you have a concrete need and have budgeted for its quirks.
- **Delivery is a distributed-systems problem in miniature.** Networks fail, endpoints time out, and re-sends can create duplicates. That's why the design has **idempotency keys, retries with backoff, and a visible delivery log** — don't skip them, or you'll get silent data loss or double-posted records.
- **Security is non-negotiable here.** Outbound pushes go only to destinations *you* configured — never a URL pulled from a document (that's an exfiltration vector). Sign payloads, keep integration secrets in a secret store, and never log them.
- **Field mapping is the hidden work.** Your schema's `total` has to land in the right column/field on the other side. Expect to build a small mapping config per destination; that's often more fiddly than the API call itself.
- **Failure UX matters.** When a delivery fails, the user needs to see it and retry — a silently dropped push is worse than no integration, because they'll assume the data arrived.

**Phasing:** (1) approve + export CSV/JSON manually → (2) generic signed webhook → (3) Google Sheets → (4) one specific CRM/ERP you actually use. Ship each rung before climbing the next.

## 14. A note on error & empty states

These feel trivial and get skipped — then the app feels broken the first time something goes wrong. The good news: they're cheap. Budget a little time to build explicit states for unsupported/oversized uploads, failed parses, "nothing extracted," empty search, empty library, empty review queue, and failed delivery. An AI agent will build these fast if you ask for them explicitly (they're in `system-architecture.md` §3.12) — just don't let them fall off the list.

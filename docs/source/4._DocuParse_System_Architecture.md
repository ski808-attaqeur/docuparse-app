## 1. High-level architecture

```mermaid
flowchart LR
    U[User / Browser] -->|HTTPS| FE[React SPA<br/>Vite + TS + Tailwind]
    FE -->|REST / SSE| API[FastAPI<br/>API service]

    API --> PG[(PostgreSQL 16<br/>+ pgvector + FTS)]
    API --> OS[(Object storage<br/>FS / MinIO)]
    API -->|enqueue| RQ[(Redis<br/>broker)]
    RQ --> W[Celery workers]

    W --> PARSE[Parse / OCR<br/>PyMuPDF, Tesseract...]
    W --> LLM[Claude API<br/>extract / Q&A]
    W --> EMB[Voyage embeddings]
    W --> PG
    W --> OS

    API -->|RAG query| LLM
    API -->|vector + FTS| PG
```

**Separation of concerns**
- **API service** вҖ” stateless request/response, auth, validation, orchestration, streaming Q&A.
- **Workers** вҖ” all heavy/slow work (parse, OCR, LLM extraction, embeddings) off the request path.
- **Postgres** вҖ” single source of truth: metadata, page text, extractions, vectors, full-text.
- **Object storage** вҖ” original binaries only.
- **Redis** вҖ” task broker + light cache.

## 2. App flows

### 2.1 Upload вҶ’ processing pipeline
```mermaid
sequenceDiagram
    participant U as User
    participant FE as React SPA
    participant API as FastAPI
    participant OS as Object Store
    participant Q as Redis/Celery
    participant W as Worker
    participant DB as Postgres

    U->>FE: Drag & drop file(s)
    FE->>API: POST /documents (multipart)
    API->>OS: store original
    API->>DB: insert document (status=queued)
    API->>Q: enqueue parse job
    API-->>FE: 202 {id, status:queued}
    loop until done/failed
        FE->>API: GET /documents/{id}/status
        API-->>FE: status
    end
    W->>Q: pick up job
    W->>OS: fetch file
    W->>W: detect type / classify (Haiku)
    W->>W: extract text (native or OCR)
    W->>DB: write pages
    W->>W: chunk + embed (Voyage)
    W->>DB: write chunks + vectors + tsv
    W->>DB: status=done, processed_at
    FE->>API: GET /documents/{id}
    API-->>FE: ready
```

### 2.2 Structured extraction + review
```mermaid
sequenceDiagram
    participant U as User
    participant FE as React SPA
    participant API as FastAPI
    participant W as Worker
    participant LLM as Claude
    participant DB as Postgres

    U->>FE: Pick schema, "Extract"
    FE->>API: POST /documents/{id}/extract {schema_id}
    API->>W: enqueue extract job
    W->>DB: read page text
    W->>LLM: tool-use call w/ JSON schema
    LLM-->>W: typed fields + confidences
    alt low confidence
        W->>LLM: escalate to Opus
    end
    W->>DB: write extraction (reviewed=false)
    FE->>API: GET /documents/{id}/extractions
    U->>FE: Review side-by-side, correct fields
    FE->>API: PATCH /extractions/{id} {corrected_data, reviewed:true}
    API->>DB: store canonical corrected data
```

### 2.5 Interactive review вҖ” bounding boxes & click-to-extract
How the two panes stay linked. This depends on **word-level bounding boxes** captured during OCR/parse and **grounded extraction** (each field carries the source span it came from).

```mermaid
flowchart TD
    subgraph Ingest
      O[OCR / PDF text layer] --> WB[Word boxes:<br/>page, x, y, w, h, text]
      WB --> DB[(store layout)]
    end
    subgraph Extract
      T[Page text] --> LLM[Claude: value + source_text span]
      LLM --> M[Match source_text вҶ’ word boxes]
      M --> F[Field: value, confidence,<br/>page, bbox rects]
      F --> DB
    end
    subgraph Review UI
      C1[Click field on right] --> HL[Highlight bbox + scroll source]
      C2[Drag box on source] --> SEL[Collect words in region]
      SEL --> FILL[Fill / correct focused field]
      FILL --> PATCH[PATCH extraction:<br/>value + user-set bbox]
    end
    DB --> C1
    DB --> C2
```

Key point: the LLM returns each field **plus the exact source substring it used**; the backend locates that substring in the stored word boxes to compute the highlight rectangle(s). Click-to-extract runs the reverse вҖ” a user-drawn region is resolved to the words inside it. (See `audit.md` for why this matching is the hard part.)

### 2.6 Approve вҶ’ deliver to integration
What happens when a reviewer hits Approve. Classification routes each document type to its mapped destination.

```mermaid
sequenceDiagram
    participant U as User
    participant API as FastAPI
    participant DB as Postgres
    participant Q as Worker
    participant D as Destination (Sheets/CRM/Webhook)

    U->>API: PATCH /extractions/{id} {status:"approved"}
    API->>DB: store approval + human corrections
    API->>DB: look up routing rule for doc_type
    alt auto_deliver_on_approve
        API->>Q: enqueue delivery (idempotency_key)
        Q->>D: signed payload (approved data)
        alt success
            D-->>Q: 2xx
            Q->>DB: delivery = success
        else failure
            D-->>Q: error / timeout
            Q->>DB: delivery = retrying (backoff)
            Q->>DB: delivery = failed (after N tries)
        end
    end
    U->>API: GET /extractions/{id}/deliveries
    API-->>U: delivery status (visible, retryable)
```

Delivery never happens silently or from a URL found inside a document вҖ” only to **user-configured** destinations, signed with HMAC. Failures surface in the UI with a retry.

**Behaviors encoded above**
- **Bidirectional link (в—„в”Җв–ә):** selecting `Invoice #`, a line-item cell, or `Total` on the right highlights and scrolls to its bounding box on the left, and vice-versa.
- **Grouping:** fields are collapsed under logical sections вҖ” *Document Header В· Line Items В· Summary* вҖ” to cut cognitive load.
- **Data table:** line items render as a spreadsheet-like grid; text columns left-aligned, numeric columns right-aligned; rows are editable and can be added/removed.
- **Confidence cues:** рҹҹў/рҹҹЎ/рҹ”ҙ plus a вҡ  flag and faded text on low-confidence fields; `в–ё Next flagged` jumps straight to the next field needing review.
- **Direct editing:** any field/cell is click-to-edit; **click-to-extract:** drag a box on the source to fill the focused field from the document text.
- **Classification shown + correctable:** the detected `Type: Invoice в–ҫ вңҺ` can be changed, which re-runs the correct schema.
- **Approve workflow:** every doc has a clear `вң“ Approve` / `Reject / Flag`; an edit summary shows what the human changed vs. the AI.

## 4. Cross-cutting concerns

- **Auth:** every route behind bearer token; SPA stores token in memory/secure cookie. No public registration.
- **Observability:** structured JSON logs; per-job timing; per-request `X-Request-Id`. Optional cost log per LLM call.
- **Failure isolation:** worker crashes/retries don't affect API availability; failed docs are visibly `failed` and retryable.
- **Security:** validate MIME server-side; sandbox/limit parser resource use; scrub file paths; never echo secrets; log all external LLM calls.
- **Data lifecycle:** deleting a document cascades to pages, chunks, vectors, and extractions; export before delete if needed. Delivery logs are retained for audit.
- **Integrations (outbound):** approved data is delivered only to user-configured destinations, HMAC-signed, with idempotency + retry + backoff; delivery status is always visible and never silent. Secrets are referenced, never stored in plain config.
- **Scaling path (Phase 2):** add worker replicas; move object storage to S3/MinIO; consider a dedicated vector store only if pgvector becomes the bottleneck.
